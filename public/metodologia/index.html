    1. Resumen ejecutivo

    Evaluamos proveedores de hosting en Chile combinando mediciones automatizadas (uptime cada hora, benchmarks Lighthouse mensuales), reputación pública (Reclamos.cl, reseñas verificadas) y curaduría editorial (verificación legal, soporte, transparencia). El ranking solo muestra empresas marcadas como verificadas y curadas, con score ≥ 7.0.

    3.b Criterios de reputación verificada

    • Reseña verificada: usuario autenticado + email confirmado + marcado is_verified_customer con prueba (factura, captura del panel o dominio activo).
    • Reclamo interno verificado: envío con email confirmado vía token de 48 h y aprobación editorial (estado verified o resolved).
    • Reclamo externo: resultados de Googlesite:reclamos.cl "proveedor"; las URLs exactas se citan en cada ficha.
    • Score reputacional: análisis automatizado mensual del texto de Reclamos.cl ponderado con la cuenta de reclamos internos verificados (1 = pésimo, 10 = excelente).

    4. Cómo se calcula el ranking

    Cómo se calculó el ranking

    Fórmula pública, datos verificables y pesos publicados.

    score_final = 0.30·reputación + 0.25·uptime + 0.20·velocidad + 0.15·soporte + 0.10·precio
    • 30%
      Reputación

      Fuente: Reclamos.cl + reseñas verificadas (hosting_reviews)

      Frecuencia: Sincronización semanal

      Normalización: 10 = 0 reclamos no resueltos último año. Resta 0.5 por cada reclamo abierto verificado.

    • 25%
      Uptime & Disponibilidad

      Fuente: Tabla uptime_pings (HEAD/GET cada hora)

      Frecuencia: Cada 1 hora · agregado a 30 días

      Normalización: uptime% mapeado: 99.99→10, 99.9→9, 99.5→7, 99.0→5, <98→0.

    • 20%
      Velocidad & Rendimiento

      Fuente: benchmark_results: TTFB mediana + Lighthouse Perf

      Frecuencia: Mensual + manual

      Normalización: 0.6 · score_lighthouse_perf/10 + 0.4 · score_ttfb (TTFB<200ms=10, >1500ms=0).

    • 15%
      Soporte & Documentación

      Fuente: Curaduría editorial + reseñas (support_rating)

      Frecuencia: Revisión trimestral

      Normalización: Promedio support_rating (1–5) escalado a 0–10 + bonus por docs en español.

    • 10%
      Precio & Valor

      Fuente: hosting_plans: relación features/precio normalizada al sector

      Frecuencia: Revisión trimestral

      Normalización: Percentil inverso del precio efectivo por GB+features. Mediana del mercado = 7.

    Reglas de exclusión

    • is_verified = false (proveedor no validado)
    • is_curated = false (sin curaduría editorial)
    • score_final < 7.0
    • Detección de monopolio o redirección a marca distinta

    Metodología de Benchmark v1.0

    Qué medimos

    Para cada proveedor de hosting curado y verificado, medimos cinco dimensiones técnicas con fuentes reproducibles:

    1. TTFB (Time To First Byte) — 5 mediciones secuenciales con fetch GET desde Supabase Edge (región us-east). Reportamos mediana y p95.
    2. Lighthouse Mobile — Vía Google PageSpeed Insights API (estrategia mobile). Categorías: Performance, SEO, Accessibility.
    3. Core Web Vitals — LCP, FCP, CLS extraídos del informe Lighthouse.
    4. Uptime 30 días — Pings horarios desde edge function. Porcentaje de respuestas OK (status 2xx/3xx) en los últimos 30 días.
    5. Tecnología — Header Server, soporte de Brotli (Content-Encoding), versión HTTP.

    URL medida

    La URL pública configurada en benchmark_target_url (normalmente la home comercial del proveedor). Esta URL es visible en cada resultado y puede ser auditada por cualquier persona.

    Score compuesto

    composite_score =
  0.35 × lighthouse_perf
+ 0.25 × ttfb_score
+ 0.25 × uptime_score
+ 0.15 × lighthouse_seo

ttfb_score   = max(0, 100 − ttfb_median_ms / 10)
uptime_score = uptime_30d_pct

    Pesos elegidos para premiar velocidad real percibida (TTFB + Lighthouse) y disponibilidad por sobre métricas cosméticas.

    Frecuencia

    • Pings de uptime: cada hora.
    • Benchmark completo: día 1 de cada mes a las 03:00 CLT, más ejecuciones manuales auditables.

    Reproducibilidad

    Cada benchmark_result guarda el JSON crudo de PageSpeed (raw_json) y las 5 muestras individuales de TTFB (ttfb_samples). Cualquier persona puede repetir las mediciones con las mismas URLs y herramientas.

    Limitaciones honestas

    • TTFB se mide desde un único origen (us-east). No representa latencia desde Chile específicamente.
    • Lighthouse usa la red simulada de PageSpeed, no una red chilena real.
    • Estamos trabajando en agregar mediciones desde Santiago (CLT) en próximas versiones.

    9. Limitaciones

    • Uptime se mide desde edge de Supabase, no replica latencia desde todos los ISP chilenos.
    • Benchmarks Lighthouse son sintéticos; no reemplazan medición real de usuarios (RUM).
    • Reclamos.cl puede contener denuncias no verificadas; las ponderamos solo cuando hay evidencia.
    • Precio se calcula sobre planes de entrada publicados; no cubre negociaciones privadas.
